# ChatGPT Web Runner — Content Script Protocol Spec

## 1. Objective
Web Runner tự động:
- mở/đảm bảo tab ChatGPT
- paste prompt (có thể nhiều phần)
- submit
- capture output
- ghi output vào Supabase (prompt_run_outputs)

Web Runner phải chịu được:
- UI ChatGPT đổi selectors
- prompt quá dài (multi-part)
- content script chưa ready (retry)
- partial capture (không mất dữ liệu)

---

## 2. Components
- UI (Preact): khởi tạo run, hiển thị progress/output
- Background: orchestrator (tab mgmt, retry, route messages)
- Content Script: thao tác DOM ChatGPT + capture

---

## 3. Message Types (UI ↔ Background ↔ Content Script)

### 3.1 Start Run
UI → Background:
- `RUN_PROMPT_START`
  - `run_id`
  - `tab_policy`: reuse_existing | open_new
  - `payload`: array of parts (string[]) OR pointer to snapshot in DB
  - `options`: { ack_gate: true, wait_ms, max_retries, selector_version }

Background → Content Script:
- `CS_RUN_PARTS_START`
  - `run_id`
  - `parts[]`
  - `ack_gate`
  - `final_instruction` (optional)
  - `timeouts`

### 3.2 Progress / Status
Content Script → Background:
- `CS_RUN_PROGRESS`
  - `run_id`
  - `phase`: preparing | sending_part | waiting_ack | waiting_response | capturing
  - `part_no`
  - `detail`

Background → UI:
- `RUN_PROMPT_PROGRESS` (forwarded)

### 3.3 Output capture
Content Script → Background:
- `CS_CAPTURE_OUTPUT_APPEND`
  - `run_id`
  - `chunk_text`
  - `order`
  - `captured_at`
  - `meta`: { selector_used, message_id_guess, partial: bool }

Background:
- writes to Supabase `prompt_run_outputs`

UI subscribes realtime to show new output rows.

### 3.4 Completion / Error
Content Script → Background:
- `CS_RUN_DONE`
  - `run_id`
  - `status`: succeeded | failed | partial
  - `error_code?`, `error_detail?`

Background → UI:
- `RUN_PROMPT_DONE`

---

## 4. Chunking / Multi-part Protocol

### 4.1 When chunking is required
Nếu payload length vượt ngưỡng 1 message:
- split into parts:
  - Part 1: rules + metrics
  - Part 2..N: ledger and appendices
  - Final: instruction "Analyze and produce output..."

### 4.2 Part format
Each part should include header:
- `### CONTEXT PART i/N`
- `run_id`, `as_of_ts`
- content payload

### 4.3 ACK Gate
Nếu `ack_gate=true`:
- Sau khi gửi Part i, chờ assistant phản hồi có chứa:
  - `ACK PART i` (hoặc pattern configurable)
- Chỉ khi ACK hợp lệ mới gửi part i+1.

Nếu `ack_gate=false`:
- gửi parts liên tục (rủi ro mất thứ tự)

### 4.4 Final instruction
Sau khi gửi parts, gửi 1 message cuối:
- “Now analyze all received parts and output according to schema …”

## 5. DOM Interaction Requirements
### 5.1 Readiness detection
Content script phải xác định:
- input box tồn tại
- send button clickable
- page is ChatGPT conversation view

### 5.2 Selector strategy
- Primary selectors (current)
- Fallback selectors (older variants)
- `selector_version` stored in meta_json

### 5.3 Submission strategy
- set textarea value (native events)
- trigger input event
- click send
- wait for assistant response blocks

### 5.4 Capture strategy
- detect new assistant message nodes
- capture incremental updates if streaming
- append outputs with ordering
- stop capture when response stable for N seconds

---

## 6. Failure Modes & Recovery
### 6.1 TAB_NOT_READY
- retry inject content script up to max_retries
- return error if still not ready

### 6.2 SELECTOR_CHANGED
- try fallback selectors
- if still fail: mark run failed with diagnostic info

### 6.3 PARTIAL_CAPTURE
- if sending succeeded but capture fails:
  - mark run partial
  - allow “Resume capture” action in UI

### 6.4 RATE_LIMIT / NETWORK
- detect UI banners or errors
- backoff and retry (bounded)
- record error in `errors` table and run meta

---

## 7. Non-negotiables
- Must write outputs incrementally to Supabase (append-only).
- Must support chunked multi-part sends.
- Must provide enough diagnostics in errors/meta to debug selector breaks.
